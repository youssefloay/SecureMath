MASTER PROJECT SPECIFICATION: Secure Math EdTech Platform
PART 1: Project Overview & Business Logic
This project is a high-security EdTech web application designed initially for a math teacher, with the architecture built to scale into a multi-teacher marketplace. The primary directive of this platform is Anti-Piracy. It is designed to sell high-value Zoom recordings while making it technically impossible for students to download, screen-record, or share access to the videos.

The 4 Core Business Rules (Strictly Enforced)
The Anti-Piracy Rule: All video playback is handled by VdoCipher, utilizing Hollywood-grade DRM (Widevine/FairPlay) and dynamic watermarking (displaying the user's email/IP on the video).

The Time Limit Rule: Once a student clicks "Play" on a purchased video, a strict 24-hour countdown begins. After 24 hours, access is permanently revoked.

The View Limit Rule: Within that 24-hour window, the student can only initiate playback a maximum of 2 times.

The "One-Screen" Policy: Similar to Netflix, a user account can only be active on one device at a time. Logging in on a new browser automatically invalidates the session on the previous browser.

The Payment Model (Egyptian Market Focus)
Payments are not processed via automated gateways like Stripe. They are handled via InstaPay / Telda through a manual verification flow:

Student receives a unique, random 6-character OTP (e.g., MATH-8291).

Student transfers money via their bank app and writes the OTP in the transfer notes.

Student uploads a screenshot of the receipt to the platform.

Admin manually verifies the screenshot against the bank app and clicks "Approve".

PART 5: Critical Logic Workflows (For AI Implementation)
Workflow 1: The "One-Screen" Policy Guard
On Login: Generate a uuidv4(). Save it to a secure HttpOnly cookie AND update the user's Firestore currentSessionId.

On Navigation (Middleware/Hook): Check if the cookie's UUID matches the Firestore currentSessionId. If they do not match, immediately execute firebase.auth().signOut(), clear the cookie, and redirect to /login?error=session_hijacked.

Workflow 2: Manual Checkout & Upload
In /checkout/[id], generate a short paymentCode using Math.random().toString(36).substring(2, 8).toUpperCase().

Use the @google-cloud/storage SDK (via a Next.js Server Action generating a signed URL) or Firebase Storage SDK to upload the student's screenshot to proofs/{userId}/{orderId}.jpg.

Create the OrderDoc in Firestore with status: "PENDING".

Workflow 3: Admin Approval & Notification
Admin views the /dashboard table showing OrderDocs where status === 'PENDING'.

Admin clicks the thumbnail to view the GCS screenshotUrl in a modal.

Admin clicks "Approve". A Server Action updates the status to APPROVED.

The Server Action immediately triggers Resend to email the student: "Your lesson is ready. Warning: You have 24 hours and 2 views starting from your first click."

Workflow 4: The VdoCipher Security Gate (/api/vdocipher/route.ts)
This API route is the vault door. It must execute these checks sequentially before returning the VdoCipher OTP to the client iframe:

Auth Check: Verify user via firebase-admin.

Ownership Check: Verify OrderDoc.userId === currentUser.uid.

Payment Check: Verify OrderDoc.status === 'APPROVED'.

Activation Logic:

If activatedAt === null: Update Firestore -> activatedAt = Date.now(), viewCount = 1.

If activatedAt !== null:

Check Time: If (Date.now() - activatedAt) > 86400000 (24h) -> Return 403 Forbidden ("Time Expired").

Check Views: If viewCount >= 2 -> Return 403 Forbidden ("View Limit Reached").

If valid: Update Firestore -> viewCount = viewCount + 1.

VdoCipher API Call: Only if all checks pass, POST to https://api.vdocipher.com/v2/otp with the video ID and return the otp and playbackInfo to the frontend.
